# 复盘：做完 DataMind 项目，我为什么收窄了 AI 的自由度？

在决定做 DataMind 这个 AI 数据分析辅助平台之前，我对大模型的能力抱有一些不切实际的幻想。

当时，市面上很多 AI 工具的做法是给用户一个对话框，支持上传文件，然后让用户自由提问。DataMind 的 v1.0 版本也是这么做的：我用 Streamlit 搭了一个简易的 MVP，加上了解析和问答功能。我原本以为，只要接上大模型，数据分析的门槛就会被彻底抹平。

但在实际测试和迭代中，我很快被现实“打脸”，并在这个过程中重新理解了产品边界。

## 1. 发现问题：面对空白对话框，业务用户其实不知道问什么

在用真实的电商数据集（sample_ecommerce.csv）做测试时，我发现了一个问题：当非技术用户面对一份原始的、未经处理的 CSV 文件时，他们不知道该问什么。

他们不知道数据里有没有异常值和缺失值，不了解数据具体情况，不知道为了看转化率，到底是该选柱状图、折线图还是漏斗图。如果直接给他们一个无约束的聊天框，他们往往只能问出“帮我分析一下这份数据”这样宽泛的问题，而大模型给出的回答往往也是非常发散、浮于表面的套话。

因此，**数据分析是有成熟的最佳实践和步骤的，不能让用户去猜，也不能让模型去盲目发散。**

## 2. 怎么改：收敛需求，强制引入 5 步工作流

为了解决这个问题，我决定调整产品的定位，把它从一个“自由对话工具”收敛为“Chat-as-a-Report（对话即报告）”的形态。

在项目的 v3.0 重构中，我引入了 LangGraph，放弃了单体单次问答，而是把复杂的分析任务强行切分成了 5 个明确的串行节点：
1. **Profiling**：先读取列名和数据类型，判断这是什么业务场景的数据。
2. **Cleaning**：处理缺失值和异常值。
3. **EDA**：做基础的统计和相关性分析。
4. **Analysis**：套用具体的业务模板（比如电商交易、营销投放等）进行深入分析。
5. **Report**：把前面的图表和结论汇总成一份 Markdown 报告。

同时，我保留了一个 Router（路由）Agent，把用户的日常提问精准分发到数据清洗、EDA、业务分析或 A/B 测试等专属的子模块中。这样的改动牺牲了一部分 AI 的“自由度”，但换来的是用户每一次上传数据，都能得到结构化、可预期的结果。

## 3. 踩过的坑与权衡：永远不要让大模型做算术题

在这个过程中，我踩过最大的坑就是大模型的“数字幻觉”。

一开始，我尝试让模型直接读取数据并总结规律，但我发现，一旦数据量稍微大一点，大模型就会开始一本正经地胡编乱造统计结果。

经过反复尝试与调整，做出了一个重要的架构决定：**彻底剥离“计算”与“推理”的职责。**

在最终的实现里，我坚决要求所有涉及缺失值诊断、RFM 建模、A/B 测试假设检验等硬核计算逻辑，全部交由 Python 原生的库（pandas、scikit-learn、scipy）来执行，可视化也交由 Plotly 渲染。**大模型在这个系统里的角色被严格限制了：它只负责解释 Python 计算出的准确数据、总结图表洞察，以及给出下一步的业务建议**。

这个 Trade-off 虽然增加了代码的复杂度（因为需要在 Agent 之间不断传递共享的数据状态），但彻底解决了数字不准确的问题。

## 4. 哪里还不够好？

DataMind 验证了“用 Agent 拆分稳定步骤、用代码保底计算”的可行性，但作为一个原型，它依然有很多不足之处，处于待拓展开发阶段：

* **模板的局限性：** 目前系统只能比较好地识别电商、营销和用户增长这几类内置的领域模板。如果用户上传一份医疗或者工业制造的数据，最后的深度分析步骤依然会显得有些泛泛而谈。
* **数据承载量：** 目前架构主要还是跑在本地和单机进程里，如果处理几十万行以上的大文件，Streamlit 的前端渲染和 Python 处理效率会遇到瓶颈，还没有经历过真实企业级大数据的考验。

做完这个项目，我最大的收获是从一个“AI 技术跟风者”，变成了一个“稍微务实一点的产品开发者”。**好用的 AI 产品不是把大模型包装一下推给用户，而是把我们的业务经验写进工作流里，在需要严谨的地方用传统代码兜底，在需要理解的地方再让 AI 上场。**
