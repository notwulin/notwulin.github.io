# 复盘：一个粗糙但跑通了闭环的 AI 竞品雷达，暂时没死磕“写爬虫”

做完 DataMind 之后，我脑子里一直有个挥之不去的痛点，现在信息都太杂乱，我之前了解到 RSS 订阅本来想做个信息聚合的东西，不过Inoreader这些也挺好用，就搁置了，但是给了我换一个角度做另一个产品的灵感，就是从商业角度上也可以利用信息聚合去了解竞品的动态（官网更新、用户吐槽、各类社区讨论）。如果能搞个系统，自动把全网信息抓过来，再让 AI 每天给我生成一份竞品的简报，俗话说得好，在商场上就是知己知彼百战不殆！

怀着这个美好的设想，我开始动手做 RivalSense（AI 竞品雷达）。

这个项目目前其实非常粗糙，尤其是在“数据获取”这一环，它离一个成熟的商产品肯定还差得十万八千里。我暂时是在这个粗糙的 MVP 里做取舍，先硬生生跑通整个系统闭环，后面有空再打磨细节。

## 1. 认清现实：放弃手写硬核爬虫，先用公开数据凑合

一开始，打算去定向抓取 Reddit 的特定节点，或者去各大社区批量拖取帖子。但我很快发现，搞定动态渲染、IP 池、验证码（CAPTCHA）和复杂的登录墙，工作量很大，不是短时间内能搞定的。

所以，我果断认怂，做了一个极其“取巧”的决定：**在这个阶段，我不针对任何特定的私有数据源写复杂的定向爬虫，只抓取最容易获取的公开信息**。

在 RivalSense 的数据引擎里，我仅仅用了最基础的 `feedparser` 和 `BeautifulSoup` 去抓取竞品的 RSS 和博客静态页。对于各大社区的声音，我甚至都没去碰它们的网站本身，而是可以选择性地调用 Tavily、Brave Search 和 SerpAPI 这些现成的结构化搜索引擎 API。换句话说，目前系统抓回来的数据其实不多，就是一点点公开的表层数据而已，但对于跑通 MVP 来说是足够的。

作为一个 PM，我当时的想法很明确：**MVP 的首要任务是跑通“获取 -> 分析 -> 报告”的整体链路，而不是把自己逼成一个反爬虫工程师。**

## 2. 核心 Trade-off：既然数据少且杂，怎么通过 Local-First 榨干价值？

虽然只抓回来一点点公开的搜索数据和资讯，但我发现里面的“废话率”依然高得惊人。大部分内容都是毫无营养的 PR 稿或者一句单薄的“Love this tool”。

如果把这些粗糙的生肉数据直接喂给大模型（Gemini 或 DeepSeek），Token 成本会原地爆炸，而且模型很容易在一堆废话中失去焦点。

为了给这批粗糙的数据“脱水”，我在大模型前面强行加了一层 **“Local-First（本地优先）”** 的 NLP 处理引擎。
我用了极其老派但稳定的库（英文用 NLTK VADER，中文用 SnowNLP），在本地先给每一条文本打分：
*   **痛点分（Pain Score）：** 匹配含有 bug、crash、too expensive 等关键词的内容。
*   **商业分（Business Score）：** 筛选融资、涨价、发布新功能等信号。

只有那些通过了本地去噪打分，并被 TF-IDF 和 KMeans 算法聚类压缩过的高价值“痛点簇”，才会被发送给大模型进行最终的摘要。这个妥协不仅极其省钱，还能让 AI 生成的简报变得特别干练。

## 3. 架构上的“缝缝补补”与闭环展示

为了快速把项目跑起来，我在后端架构上也留下了不少“凑合”的痕迹。

要是懂行的老开发看一眼我的后端源码，估计会笑出声。我没有上那些高大上的分布式队列（比如 Redis + Celery），而是直接把抓取任务（Job）塞在了 Flask 的进程内存里跑后台线程（ThreadPoolExecutor）。这就导致一个极其尴尬的问题：只要后端服务一重启，旧的任务 ID 就会全部失效，变成一堆死数据。同时，数据库我也只用了最轻量的 SQLite，开个 WAL 模式勉强撑一下并发读写。（我也不是纯码农，先凑合一下）。

但我为什么觉得这个项目依然有价值？因为我**极其重视它的展示闭环**。

我知道现在的朋友没有耐心去等我那粗糙的后端慢吞吞地爬数据。所以我在 Next.js 的前端里专门做了一个 `NEXT_PUBLIC_DEMO_MODE`（演示模式）。只要开启这个模式，前端就会直接加载我内置好的 Mock 采样数据，秒开网页，直接模拟出任务加载的进度条和最终生成的 AI 报告看板。

## 4. 下一步：从“跑通闭环”到“真正可用”

虽然 RivalSense 勉强跑通了全链路，但我也非常清楚，它现在的状态离一个生产级产品还差得很远：
1.  **数据抓取的重构：** 这是最大的优化空间。下一步必须老老实实对接合法的外部 API（比如 Trustpilot 或 G2 的商业接口），或者建立爬虫代理池，针对高价值源（如特定的 Reddit 节点）写深度解析器。
2.  **后端的工程化：** 必须把 SQLite 换成 PostgreSQL，把内存里的任务队列迁移到外部的可靠队列里去，解决服务重启导致任务丢失的硬伤。

做完这个项目，虽然它现在看起来还有点像个用规则拼接起来的“玩具”，但我其实挺有成就感的。**好钢要用在刀刃上。作为一个 PM 视角的技术探索者，优先搭出包含前端交互、本地去噪和 AI 摘要的骨架，比在一个死胡同里死磕反爬机制要重要得多。** 哪怕产品糙一点，至少它“活着”跑通了。
